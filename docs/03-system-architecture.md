# 系统架构设计（V1）

## 职责
- 定义系统技术架构与关键技术链路，作为研发实现基线。

## 包含内容
- 逻辑分层
- 技术组件职责
- 关键业务链路
- 部署建议

## 不包含内容
- 详细表结构
- 具体接口字段定义
- 细粒度需求验收标准

---

## 一、总体架构
- 前端管理端 -> NestJS API -> 业务模块服务 -> PostgreSQL / Redis / MinIO

## 二、技术组件职责

### 1) NestJS（业务服务层）
- 负责认证鉴权、业务规则、状态流转、审计记录
- 模块化拆分：auth、org、exam、marking、score、analysis、file、audit

### 2) PostgreSQL（事务数据）
- 存储用户、档案、考试、阅卷任务、成绩、日志等核心结构化数据

### 3) Redis（缓存与临时状态）
- 会话与令牌辅助
- 验证码/限流/短期任务状态
- 热点查询缓存

### 4) MinIO（对象存储）
- 存储模板文件、导入文件、答题卡图片、导出结果等非结构化文件
- 通过对象Key与业务数据关联

## 三、关键业务链路

### 1) 登录链路
- 用户登录 -> 校验身份 -> 生成令牌 -> 角色权限生效

### 2) 建考到阅卷链路
- 创建考试 -> 配置考试科目与班级 -> 生成阅卷任务 -> 教师提交阅卷 -> 汇总成绩

### 3) 发布与分析链路
- 管理员发布成绩 -> 固化发布状态 -> 提供考试统计分析

### 4) 文件链路
- 前端上传 -> 后端写入MinIO -> 返回对象标识 -> 数据库记录对象关系

## 四、部署建议（MVP）
- 采用 `docker compose` 部署以下服务：
  - `api`（NestJS）
  - `postgres`
  - `redis`
  - `minio`
- 可选 `nginx` 作为统一入口与反向代理。

## 五、非功能性要求（基础）
- 安全：密码哈希存储、最小权限、接口鉴权、操作留痕
- 可用性：关键流程异常可追踪（日志与错误码）
- 可扩展性：模块边界清晰，可逐步拆分服务

## 六、本地开发（仓库已实现）

- **依赖服务**：仓库根目录 `docker-compose.yml` 提供 PostgreSQL、Redis、MinIO；启动：`docker compose up -d`。
- **环境变量**：复制根目录 `.env.example` 为 `.env`（勿提交），其中 `DATABASE_URL` 需与 compose 中 Postgres 一致。
- **数据库迁移**：在 `apps/api` 目录执行 `npx prisma migrate dev`（开发）或 `npx prisma migrate deploy`（CI/生产）。
- **API 服务**：在 `apps/api` 执行 `npm run start:dev`；全局路由前缀默认 `api/v1`（环境变量 `API_PREFIX`）；就绪检查：`GET /api/v1/health`（依赖数据库已迁移且可用）。
