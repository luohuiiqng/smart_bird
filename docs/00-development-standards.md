# 开发与界面规范（生产级）

## 职责
- 约束本仓库全栈开发的底线标准：**禁止推测性实现与写死业务逻辑**，并与**学校场景 UI** 取向对齐。
- 作为编码、评审、排期的统一依据；与具体模块规格文档配合使用。

## 包含内容
- 代码与配置层面的生产级要求（禁止“猜接口、猜分支、魔数业务规则”）。
- 配置、错误处理、安全、可观测性等底线。
- 管理端界面视觉与交互取向（学校风格）。

## 不包含内容
- 各业务模块的接口字段明细（见对应 `06`～`11` 文档）。
- 具体组件库选型变更细节（选型应在架构文档中变更）。

---

## 一、禁止推测与写死逻辑

### 1) 禁止“推测实现”
- 业务规则、接口契约、状态流转必须以 `docs` 中规格文档为准；文档未定义的行为不得擅自补齐。
- 若需求不明：**停止编码**，先更新规格文档或拉齐确认，再实现。
- 禁止依赖未文档化的第三方接口行为假设。

### 2) 禁止写死业务逻辑
- **禁止**在代码中硬编码学校 ID、角色名映射、科目列表、及格线、分页默认值等业务含义（除非规格明确规定且集中在单一配置源）。
- **禁止**用魔法字符串散落判断状态；状态枚举须类型化并在规格文档中有定义。
- **禁止**用假数据、占位返回值冒充真实逻辑（开发环境可用明确标注的 mock，且不得进入生产构建路径）。

### 3) 配置与默认值
- 环境相关项（数据库、Redis、MinIO、JWT、域名）必须从环境变量或配置中心读取；仓库内仅保留示例模板（`.env.example`），**不得提交真实密钥**。
- 业务默认值（如分页、及格率阈值）若在规格中未定稿，应在规格文档中定义后再实现；实现时集中放在配置模块并可在部署时调整。

---

## 二、生产级工程要求（后端）

- **校验**：所有外部输入使用 DTO + 校验（class-validator 等），拒绝静默吞错。
- **错误**：统一错误结构、明确错误码；禁止向客户端泄露堆栈与内部路径。
- **事务**：跨表写操作使用数据库事务；并发场景配合乐观锁或状态机校验。
- **鉴权**：接口默认鉴权；按学校维度隔离数据；敏感操作写审计日志（见 `11-files-audit-module-spec.md`）。
- **可观测性**：结构化日志、请求关联 ID；关键业务节点记录审计事件。
- **测试**：核心领域逻辑具备单元测试；关键接口具备契约或集成测试（按里程碑逐步补齐）。

---

## 三、生产级工程要求（前端）

- **类型安全**：与后端契约对齐（生成类型或手写 DTO），禁止 `any` 泛滥。
- **状态与错误**：接口失败须可感知（Toast/错误页），禁止静默失败。
- **无障碍**：表单控件具备标签与焦点顺序；关键操作需二次确认（删除、发布、撤回）。
- **性能**：列表分页与骨架屏；大表虚拟滚动按里程碑引入。

---

## 四、学校风格 UI 取向

目标：**清晰、稳重、可信**，符合教育管理场景，避免娱乐化与过度装饰。

### 1) 视觉
- **色彩**：以白/浅灰背景为主；主色建议深蓝或教育绿系（具体色板在实现阶段落到设计 token，禁止页面内随意硬编码色值）。
- **层次**：卡片分区、适度阴影；信息密度适中，避免拥挤。
- **字体**：系统字体栈或开源中文无版权纠纷字体；标题与正文层级分明。

### 2) 布局与组件
- **布局**：后台常规「顶栏 + 侧栏 + 内容区」；侧栏可折叠；面包屑表达层级。
- **表格**：斑马纹或行间 hover；固定操作列；空状态有引导文案。
- **表单**：标签对齐一致；必填显式标注；校验提示贴近字段。

### 3) 一致性
- 间距、圆角、按钮规格统一由设计 token / 组件封装体现；禁止同一系统多套按钮风格并存。
